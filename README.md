# Projeto Imoveis

Data pipeline for collecting, normalizing, deduplicating, and maintaining a
historical store of real estate sale and rental listings. The project collects
data from OLX, QuintoAndar, and Lopes, converts records into a canonical schema,
and updates an incremental historical database ready for analysis.

## Overview

The current flow separates collection into four stages:

1. `collect_discovery`: discovers listing URLs by source.
2. `collect_listings`: visits discovered URLs and collects complete listing data.
3. `build_daily_snapshot`: normalizes, enriches, deduplicates, and builds the daily snapshot.
4. `update_historical_store`: upserts the snapshot into the historical store.

Generated data is written to `raw/`, `processed/`, `artifacts/`, and `logs/`.
These directories are generated by pipeline runs and are not versioned in Git.

## Supported Sources

| Source | Discovery | Listings | Incremental rule |
| --- | --- | --- | --- |
| OLX | `scrapers/olx_discovery.py` | `scrapers/olx_listings.py` | new listings or listings with changed prices |
| Lopes | `scrapers/lopes_discovery.py` | `scrapers/lopes_listings.py` | new listings or listings with changed `lastmod` |
| QuintoAndar | `scrapers/quinto_discovery.py` | `scrapers/quinto_listings.py` | new listings or listings with changed `lastmod` |

Sources are registered in `scrapers/registry.py`.

## Installation

Main requirements:

- Python 3.13 or another version compatible with the project dependencies
- `pip`
- Network access for the scrapers

Install dependencies:

```sh
python -m pip install -r requirements.txt
```

## Auxiliary Brazilian ZIP Code Datasets

ZIP code enrichment uses auxiliary files in `artifacts/_cache/`. The expected
directory is:

```text
artifacts/_cache/
  base_ceps.csv
  cepaberto.csv
  zipcode_enrichment.json
```

`base_ceps.csv` should be obtained from the Base dos Dados ZIP code table:

```text
https://basedosdados.org/dataset/33b49786-fb5f-496f-bb7c-9811c985af8e?table=566ed7f2-db5f-4ac3-bfce-b579137c993c
```

The recommended collection method is Google BigQuery, exporting the result to a
single CSV named `base_ceps.csv`.

`cepaberto.csv` should be prepared from CEP Aberto data:

```text
https://www.cepaberto.com/
```

CEP Aberto provides files separated by state. The recommendation is to merge all
of them into a single CSV named `cepaberto.csv` before running the pipeline.

Both CSVs should preserve, when available, the columns used by the enricher:

```text
cep, logradouro, localidade, nome_municipio, sigla_uf, centroide
```

`zipcode_enrichment.json` is the incremental cache generated by the pipeline
itself.

## How To Run

`run-all` and `run-stage` require `--output-path`. This path is the persistent
root for `raw/`, `processed/`, `artifacts/`, and `logs/`. Use `--output-path .`
to preserve the local layout, or point it to persistent storage in a remote
environment.

List available stages:

```sh
python imoveis_pipeline.py list-stages
```

Run the full pipeline for the current day:

```sh
python imoveis_pipeline.py run-all --output-path .
```

Run continuously once per day. The first run starts immediately; the next runs
are scheduled at a random local time inside the daily start window. If no window
is provided, the default is `09:00-15:00`.

```sh
python imoveis_pipeline.py run-all --output-path /dados/projeto-imoveis --daily
python imoveis_pipeline.py run-all --output-path /dados/projeto-imoveis --daily --start-window 10:00-16:00
```

Run the full pipeline for a specific date:

```sh
python imoveis_pipeline.py run-all --output-path . --date DD-MM-YYYY
```

Run with more detailed scraper logs:

```sh
python imoveis_pipeline.py run-all --output-path . --date DD-MM-YYYY --verbose
```

Force a new discovery even when a successful manifest already exists:

```sh
python imoveis_pipeline.py run-all --output-path . --date DD-MM-YYYY --force-discovery
```

Resume the pipeline from a specific stage:

```sh
python imoveis_pipeline.py run-all --output-path . --date DD-MM-YYYY --from-stage collect_listings
```

Run a single stage:

```sh
python imoveis_pipeline.py run-stage collect_discovery --output-path . --date DD-MM-YYYY
python imoveis_pipeline.py run-stage collect_listings --output-path . --date DD-MM-YYYY
python imoveis_pipeline.py run-stage build_daily_snapshot --output-path . --date DD-MM-YYYY
python imoveis_pipeline.py run-stage update_historical_store --output-path . --date DD-MM-YYYY
```

Limit a single stage to specific sources:

```sh
python imoveis_pipeline.py run-stage collect_discovery --output-path . --date DD-MM-YYYY --sources olx lopes
python imoveis_pipeline.py run-stage collect_listings --output-path . --date DD-MM-YYYY --sources olx lopes
```

When `--sources` is used, scoped manifests and outputs may be written under
`sources__<source>` subdirectories.

`processed/<date>/` is the daily snapshot. The root `processed/` directory
contains the latest historical version.

## Incremental Behavior

- `collect_discovery` uses source-specific rules to reduce collection to records
  that are new or changed.
- `run-all` reuses an existing successful `collect_discovery` manifest for the
  date unless `--force-discovery` is provided.
- If discovery finishes with no new links, the pipeline exits early with
  `stop_reason = no_new_links_after_discovery`.
- `collect_listings` uses resume state in `artifacts/<date>/collect_listings` to
  avoid repeating completed work.
- Stages that depend on previous stages automatically read the previous stage
  manifest unless `--input-manifest` is provided.

## First Run By Source

The first run of `collect_discovery` and `collect_listings` can take very
different amounts of time by source. Discovery is usually fast; complete listing
collection is the slow stage.

On OLX, discovery tends to return only recent listings for the day, so the first
collection is usually smaller. For Lopes and QuintoAndar, discovery traverses
the historical listings available on the platforms; because of that, the first
run of those two sources can take several days to complete the initial listings
backlog.

While the first Lopes and QuintoAndar collections are still running, it is
recommended to run OLX separately with `--sources`:

```sh
python imoveis_pipeline.py run-stage collect_discovery --output-path . --date DD-MM-YYYY --sources olx
python imoveis_pipeline.py run-stage collect_listings --output-path . --date DD-MM-YYYY --sources olx
python imoveis_pipeline.py run-stage build_daily_snapshot --output-path . --date DD-MM-YYYY --sources olx
python imoveis_pipeline.py run-stage update_historical_store --output-path . --date DD-MM-YYYY --sources olx
```

After Lopes and QuintoAndar complete their first collection, the pipeline can be
run normally again for all sources.

## Data Model

The daily snapshot generates three main tables:

- `listings_unificados`: listings normalized by source, business type, and
  listing identifier.
- `properties_unified`: deduplicated canonical properties.
- `property_listing_link`: relationship between listings and canonical
  properties.

The canonical listings schema is defined in `pipelines/normalize.py` as
`CANONICAL_COLUMNS`. Long text fields such as `description` and
`long_description` are used during processing, but removed from the final
listings output by the projector in `pipelines/daily_snapshot.py`.

The historical store upserts by:

```text
source | business_type | property_id
```

and adds history metadata:

```text
first_seen_at, last_seen_at, created_at, updated_at
```

## Utilities

Export Parquet to CSV:

```sh
python parquet_to_csv.py --date DD-MM-YYYY
python parquet_to_csv.py --input processed/listings_unificados.parquet --output processed/listings_unificados.csv
```

Recover missing QuintoAndar listings for a specific date:

```sh
python workaround_quinto_missing.py paths --date DD-MM-YYYY
python workaround_quinto_missing.py build-missing --date DD-MM-YYYY
python workaround_quinto_missing.py collect-missing --date DD-MM-YYYY
python workaround_quinto_missing.py merge --date DD-MM-YYYY
```

ZIP code enrichment is handled by `pipelines/zipcode_enrichment.py`; prepare the
auxiliary datasets described in the Auxiliary ZIP Code Datasets section before
running the pipeline.

## Data Versioning

The following directories are ignored by Git:

```text
raw/
processed/
artifacts/
logs/
```

They may contain large files and reproducible outputs.

## Troubleshooting

- `manifesto de entrada nao encontrado`: run the previous stage for the same
  date or pass `--input-manifest`.
- Existing discovery was reused: use `--force-discovery`.
- Complete collection was interrupted: run the same stage/date again; resume
  state in `artifacts/<date>/collect_listings` should avoid repeating completed
  items.
